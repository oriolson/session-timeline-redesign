00 / Context

The Problem

When a user opens a Copilot agent session on iOS, they see a flat list of every event — every tool call, every intermediate reasoning message, every file read. On a 30+ tool call session, the answer is buried under dozens of rows. There's no distinction between "what happened" and "what matters."

Before

30+ rows

Everything visible, nothing scannable

After

5–6 rows

Tool calls collapsed into work bursts

03 / Architecture

How It Works

The event processing pipeline. New code enters at step 3 — everything else is existing infrastructure.

1
Raw Events
Server-sent events arrive: tool call starts, tool call completions, assistant messages, permissions.
2
makeSessionItemsAndMetadata()
Event loop pairs tool start/complete events into ToolCall objects. flushTools() emits small groups (1–2 tools) because assistant messages between tool calls trigger flushes. Keep this simple.
3
groupToolCallBursts() ← NEW
Post-processing step. Scans the final array left-to-right and groups 3+ consecutive tool calls into burst headers. Absorbs intermediate assistant messages as "thinking" items inside bursts.
4
Auto-expand last burst ← NEW
One-liner after grouping. If the last item is a burst, set isExpanded = true. This ensures users always see the most recent work.
5
AgentSessionSection.viewModels
Builds [ViewModelBinder] array for the UITableView. makeToolCallViewModels(for:) handles burst headers + expanded children using ToolCallEventCell — not ToolCallEventGroupItemCell.
6
UITableView via Adapter
UIHostingConfiguration wraps SwiftUI views in UIKit cells. .margins() controls cell-level spacing. The adapter renders ViewModelBinder array into rows.

04 / Lessons Learned

Gotchas

Non-obvious bugs and dead ends hit during prototyping. These will save you time.

⚡
Grouping in flushTools() doesn't work ▸
What we tried

Grouping tool calls inside flushTools() in the event loop.

What went wrong

Assistant messages between tool calls trigger flushTools(), so each flush only contains 1–2 tool calls. You never get groups of 3+.

The fix

groupToolCallBursts() runs as a post-processing step on the final [AgentSessionCellViewModel] array. This is the only place where you can see the full sequence.

⚡
All bursts auto-expand (not just the last one) ▸
What we tried

isExpanded = status == .running on burst headers.

What went wrong

Some tool calls in completed sessions never received completion events, so their status stayed .running. ALL bursts expanded.

The fix

Default all bursts to isExpanded = false. Then, as a separate step after grouping, set isExpanded = true only on the very last item if it's a burst.

if let lastIndex = items.indices.last, case .toolCall(var vm) = items[lastIndex], vm.groupChildren != nil { vm.isExpanded = true items[lastIndex] = .toolCall(vm) }
⚡
Final assistant message gets swallowed by burst ▸
What we tried

flushBurst() that creates the burst group and emits pending messages.

What went wrong

When the burst was the last thing in the array, pendingMessages (the final response) were silently dropped.

The fix

flushBurst() must ALWAYS emit pendingMessages after the burst. Intermediate assistant messages → absorbed as thinking items. The last assistant message with content → stays outside as the response. See the lastAssistantIndex logic.

⚡
Expanded children render as mono text, not icon rows ▸
What we tried

Used ToolCallEventGroupItemCell for expanded burst children.

What went wrong

That cell renders as plain monospace text (the old grouped-by-name style). Users expected icon + title format.

The fix

Use ToolCallEventCell for expanded children — same cell type as standalone tool calls. Each child gets its proper icon, title, and tap-to-view-output behavior.

⚡
Thinking messages leak outside bursts ▸
What we tried

Only absorbing messages with reasoning field and empty content.

What went wrong

Some assistant "thinking" messages put their text in content rather than reasoning. The narrow check missed them.

The fix

Absorb ALL pending assistant messages into the burst, EXCEPT the last one with content. Use index-based detection:

let lastAssistantIndex = pendingMessages.lastIndex { item in if case .message(let msg) = item, msg.role == .assistant, !msg.content.isEmpty { return true } return false } // Before lastAssistantIndex → thinking item // lastAssistantIndex → regular message outside burst
